Kernel Project

A GitHub Actions CI/CD pipeline that compiles, packages, and publishes Android Linux 5.15 kernels for Qualcomm MSM devices. Three build workflows handle everything from automated weekly releases to fully customized one-off builds.

The Three Build Workflows

🤖 Automated · Weekly

YASK

Fully automated weekly release pipeline. Builds all four kernel variants (GKI + CLO, KSU + non-KSU) and publishes them as a GitHub Release with a Telegram announcement.

4 variants LTO full canary KSU
🚀 Manual · Simple

Main Build

Quick manual build from any kernel source and branch. Pick KSU, non-KSU, or both variants. Fixed thin LTO. Delivers a ZIP to Telegram. Good for testing custom forks.

noksu/ksu/both LTO thin stable KSU
⚙️ Manual · Full Control

Advanced Build

Exposes every major build variable: LTO mode, optimization level, tick rate, and toolchain. Two toolchain modes: AOSP android-tools or custom greenforce_clang.

O2/O3/Os 100–1000 Hz thin or full LTO

Shared Infrastructure

📦

AnyKernel3 packaging

All three workflows produce a flashable AnyKernel3 ZIP. YASK uses the dedicated YASK branch; Main and Advanced use master.

🩹

Shared patch set

All workflows apply the same patches from topnotchfreaks/kernel_patches: sign-builds, no-dirty suffix, no-auto_localversion, and the GKI fixup patch.

🔧

Swap expansion

GitHub runners get 10–20 GB swap via pierotofy/set-swap-space at vm.swappiness=80 to handle full-LTO kernel builds without OOM.

📱

Telegram notifications

All three workflows notify on success and failure. YASK sends a photo + download link; Main and Advanced attach the ZIP file directly as a document.

Kernel Features

⚡

Full & Thin LTO

Clang Link-Time Optimisation. YASK ships with LTO=full; Main uses LTO=thin; Advanced lets you choose at runtime.

🌐

TCP Congestion Control

BBR and Westwood enabled. Westwood set as default — better for mobile/LTE links than CUBIC.

🛡️

Baseband Guard (BBG)

LSM module protecting the modem/baseband subsystem. Registered alongside SELinux. In YASK (KSU builds) and Advanced (all builds).

🔑

Wild KernelSU

Actively maintained KSU fork. YASK and Advanced use canary; Main uses stable. All include full SUSFS integration.

🫥

SUSFS Root Hiding

Kernel-level root detection bypass: magic mount, uname/cmdline spoofing, sus path/mount hiding, kallsyms symbol removal, and open redirect.

🗂️

OverlayFS + TmpFS

OverlayFS, TmpFS XATTR, and POSIX ACL for systemless module compatibility with Magisk, KSU modules, and APATCH.

🔎

Full KALLSYMS

Complete kernel symbol table — required by Metamodule Magisk modules and kernel-level debugging tools.

📝

Signed Builds

The sign-builds patch ensures kernel images pass boot verification. Applied by all three workflows.

YASK — Automated Weekly Release

schedule · sunday 00:00 UTC workflow_dispatch 4-target matrix LTO full Wild-KSU canary

YASK (Yet Another Stock Kernel) is the flagship automated workflow. It runs every Sunday midnight UTC, builds all four kernel variants in parallel, packages them into a single AnyKernel3 ZIP, pushes it to the rolling YASK-NEXT GitHub Release, and sends a Telegram photo notification with build metadata and a direct download link.

✅
YASK is the recommended way to get the latest kernel for end users. The release is always at github.com/topnotchfreaks/<repo>/releases/tag/YASK-NEXT.

Trigger

YASK.yml · trigger block
on:
  schedule:
    - cron: '0 0 * * 0'   # Every Sunday 00:00 UTC
  workflow_dispatch:     # Triggerable manually from the Actions tab

Pipeline — build-kernels job

Setup workspace
Creates the matrix-specific directory, stamps BUILD_TIME env var, installs repo rsync aria2.
Setup SWAP 10 GB
Expands GitHub runner swap to 10 GB via pierotofy/set-swap-space with vm.swappiness=80.
Clone kernel source
Shallow clones the matrix kernel repo (--depth=1 --recursive) into its dedicated directory.
Initialize AOSP repo + toolchains
Syncs the AOSP kernel manifest, downloads clang-r547379 via aria2c (16 connections), removes three unused Clang versions to save disk, patches build.config.constants with BRANCH and CLANG_VERSION, removes the defconfig integrity check.
Configure kernel & patches
Appends CONFIG_LOCALVERSION using the matrix local_version + build date. Applies the GKI fixup patch. Loops through per-matrix extra_patches (sign-builds, no-dirty, no-auto_localversion).
Add BBG — KSU variants only
Runs the Baseband Guard setup script, appends CONFIG_BBG=y, registers baseband_guard alongside SELinux as a default LSM in security/Kconfig.
Setup Wild-KSU + SUSFS — KSU variants only
Installs Wild_KSU canary. Clones SUSFS4KSU, applies the kernel patch, force-copies fs/susfs.c, include/linux/susfs.h, susfs_def.h. Integrates susfs_init() into Wild_KSU/kernel/ksu.c via patch with sed fallback. Runs full verification — hard-fails if anything is missing. Appends all SUSFS Kconfig options.
Basic configs
Appends performance, TCP congestion, OverlayFS, TmpFS, and KALLSYMS options to the defconfig.
Patch Makefile
Adds KBUILD_CFLAGS += -Wno-implicit-function-declaration to suppress upstream warnings that would abort the build.
Build kernel
Calls build/build.sh with LTO=full using all available CPU cores.
Upload artifact
Uploads the built Image as a GitHub Actions artifact (1-day retention) for the packaging job.

Pipeline — package-kernels job

Download all artifacts
Downloads all four Image files from the parallel matrix jobs.
Create release package
Clones AnyKernel3 YASK branch, copies all four images, writes ASCII banner, zips into Anykernel3_YASK-WildKSU-AIO-FULL_YYYY-MM-DD.zip.
Upload to GitHub Release
Pushes ZIP to the rolling YASK-NEXT release tag via softprops/action-gh-release. Requires the GH_TOKEN secret.
Telegram notification
On success: sendPhoto with the YASK banner, build metadata, supported devices, and download link. On failure: sendPhoto with failure details and a run log link.

Main Build — Quick Manual Build

workflow_dispatch only LTO thin Wild-KSU stable clang-r547379

The simplest manual workflow. Accepts a kernel source URL and branch, builds KSU and/or non-KSU variants, and delivers a ZIP to Telegram. Ideal for quickly testing upstream branches or a custom fork without touching the automated YASK pipeline.

💡
Main uses Wild-KSU stable — making it the right choice when you need a production-ready, tested KSU build rather than the bleeding-edge canary used by YASK and Advanced.
⚠️
Main Build does not include Baseband Guard (BBG). Use YASK or Advanced Build if BBG is required.

Pipeline Steps

Setup workspace
Installs repo rsync aria2, sets BUILD_TIME.
Setup SWAP 10 GB
Same configuration as YASK.
Clone kernel source
Clones from the kernelSourceURL and kernelBranch workflow inputs.
Initialize repo
Syncs AOSP manifest, installs clang-r547379, removes old Clangs, patches build.config.constants, replaces the hardcoded gki_defconfig in build.config.gki with the user device defconfig, disables KMI enforcement in build.config.gki.aarch64.
Configure kernel
Appends optional CONFIG_LOCALVERSION, CONFIG_KALLSYMS, CONFIG_TMPFS_XATTR, CONFIG_TMPFS_POSIX_ACL.
Setup Wild-KSU + SUSFS — ksu variant only
Installs Wild_KSU stable. Applies SUSFS kernel patch, force-copies source files, creates drivers/kernelsu → ../KernelSU symlink, appends SUSFS Kconfig options. No BBG step.
Build kernel
Calls build/build.sh with LTO=thin.
Package + Telegram
Clones AnyKernel3 master, zips as Anykernel3_WildKSU_<branch>_<date>.zip, uploads artifact (7 days), sends ZIP directly to Telegram as a document.

Advanced Build — Full Control

workflow_dispatch O2 / O3 / Os 100–1000 Hz 2 toolchain modes thin or full LTO

The most powerful workflow. Every significant build parameter is exposed as a GitHub Actions input. Supports two toolchain modes and provides the most detailed Telegram build summary.

Toolchain Modes

ModeClangSourceBuild Command
android-toolsr584948btopnotchfreaks/clangbuild/build.sh via AOSP repo
custom-toolsgreenforce latestgreenforce-project/greenforce_clangmake LLVM=1 CC=clang directly

custom-tools fetches the latest release from the greenforce GitHub API at build time. android-tools uses pinned AOSP Clang r584948b for reproducible builds.

Pipeline Steps

Setup workspace
Installs extended packages: build-essential bc curl flex cpio dwarves libelf-dev gcc-aarch64-linux-gnu libssl-dev python3 clang lld llvm in addition to the base set.
Initialize repo (android-tools only)
Installs clang-r584948b, removes more old Clangs including r547379, fully disables KMI enforcement including MODULES_ORDER and MODULES_LIST.
Get Toolchains (custom-tools only)
Queries GitHub releases API for the latest greenforce_clang, downloads, extracts, exports CC AR AS NM OBJCOPY OBJDUMP STRIP, validates clang is in PATH.
Setup SWAP 20 GB
Doubled to 20 GB vs 10 GB in YASK/main to handle O3 + full-LTO without OOM.
Patch Makefile
Adds four warning suppressions — one more than YASK/main: adds -Wno-default-const-init-unsafe, -Wno-uninitialized-const-pointer, -Wno-implicit-function-declaration, -Wno-sometimes-uninitialized.
Configure kernel
Handles LTO (thin/full with explicit Kconfig entries), optional localversion, KALLSYMS/TmpFS, optimization level via CONFIG_CC_OPTIMIZE_FOR_*, tick rate (removes conflicting HZ entries, sets new ones), and applies the sign-builds patch.
Add BBG — all builds
Unlike YASK, Advanced runs the BBG setup unconditionally — all builds get Baseband Guard regardless of KSU variant.
Setup Wild-KSU + SUSFS — ksu variant
Installs Wild_KSU canary. Applies SUSFS patches, force-copies source files. Creates drivers/kernelsu → ../KernelSU symlink (note: KernelSU, not Wild_KSU as in YASK).
Build & Package
android-tools: build/build.sh with LTO flag. custom-tools: make defconfig then make LLVM=1. Packages into AnyKernel3 master branch ZIP, artifact for 1 day. Telegram document with full config metadata in caption.
⚠️
Known bug: The LTO input is declared as LTOOptimizations (capital O) but referenced in the build step condition as LTOoptimizations (lowercase o). This mismatch may cause the thin path to always be used regardless of what you select.
💡
Advanced uses clang-r584948b in android-tools mode — newer than the r547379 used by YASK and Main. This may produce slightly different code generation between builds.

KernelSU & SUSFS Integration

✅
All three workflows use Wild_KSU combined with SUSFS for kernel-level root detection bypass.

Wild_KSU Installation

# Canary — YASK.yml and advanced_build.yml
curl -LSs "https://raw.githubusercontent.com/WildKernels/Wild_KSU/wild/kernel/setup.sh" | bash -s canary

# Stable — main.yml
curl -LSs "https://raw.githubusercontent.com/WildKernels/Wild_KSU/wild/kernel/setup.sh" | bash -s stable

SUSFS Integration Process

1 · Apply kernel patch
50_add_susfs_in_gki-android13-5.15.patch hooks SUSFS into core VFS. If this fails, source files are still force-copied in the next step.
2 · Copy source files
fs/susfs.c, include/linux/susfs.h, and include/linux/susfs_def.h are always force-copied from SUSFS4KSU (gki-android13-5.15 branch).
3 · Integrate susfs_init() into ksu.c — YASK only
YASK additionally applies 10_enable_susfs_for_ksu.patch to inject susfs_init() into Wild_KSU's init. Falls back to sed insertion. Build is hard-failed if the include or call is missing.
4 · Append Kconfig options
All 15 SUSFS feature flags are appended to the defconfig.

Secrets & Repository Tokens

⚠️
Configure these under GitHub → Repository → Settings → Secrets and variables → Actions → New repository secret before running any workflow.
GH_TOKEN
A GitHub Personal Access Token with repo scope. Required only by YASK.yml to push the ZIP to the YASK-NEXT GitHub Release via softprops/action-gh-release. Not needed for Main or Advanced.
TELEGRAM_BOT_TOKEN
Telegram Bot API token from @BotFather. Used by all three workflows — YASK via sendPhoto, Main and Advanced via sendDocument.
TELEGRAM_USER_ID
Telegram chat ID of the recipient (personal user ID or channel ID — bot must be admin for channels). Used by all three workflows. Obtain via @userinfobot.
