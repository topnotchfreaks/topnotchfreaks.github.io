The Three Build Projects

All three projects share the same kernel sources, toolchains, and patches — but differ in how they're triggered, how configurable they are, and what they produce.

Auto · Weekly

YASK

Fully automated weekly release pipeline. Builds all four GKI and CLO kernel variants (KSU and non-KSU) and publishes them as a GitHub Release with a Telegram announcement.

4 variants LTO full
Manual · Simple

Main Build

Quick manual build from any kernel repo and branch. Choose to build KSU, non-KSU, or both. Fixed thin LTO. Good for testing custom kernel forks.

noksu/ksu/both LTO thin
Manual · Full Control

Advanced Build

Exposes every major build option — optimization level, tick rate, LTO mode, toolchain choice. Two toolchain modes: AOSP android-tools or custom greenforce_clang.

O2/O3/Os 100–1000Hz

Shared Infrastructure

📦

AnyKernel3 packaging

All builds produce a flashable AnyKernel3 ZIP. YASK uses the YASK branch; Main and Advanced use the master branch.

🛠️

Shared patches

All three projects apply the same upstream patches from topnotchfreaks/kernel_patches: sign-builds, no-dirty, no-auto_localversion, GKI fixup.

🤖

GitHub Actions

Everything runs on ubuntu-latest runners. Swap is expanded (10–20 GB) to handle the memory demands of kernel compilation with LTO.

📱

Telegram notifications

All three notify on success and failure via Telegram Bot API — YASK sends a photo with a download link; Main and Advanced send the ZIP file directly.

Kernel Features

⚡

Full & Thin LTO

Clang Link-Time Optimisation reduces binary size and improves runtime performance. YASK uses LTO=full; Main uses LTO=thin; Advanced lets you choose.

🌐

TCP Congestion Control

BBR and Westwood enabled with Westwood as default. Westwood is better suited for mobile/LTE links; BBR is available for high-throughput scenarios.

🛡️

Baseband Guard (BBG)

An LSM module that protects the modem/baseband subsystem. Injected into all KSU builds and registered in the kernel's LSM chain alongside SELinux.

🔑

Wild KernelSU

A maintained KSU fork with active development. YASK and Advanced use canary builds; Main uses stable.

🫥

SUSFS Root Hiding

Full SUSFS integration: magic mount, uname spoofing, cmdline spoofing, sus path/mount hiding, open redirect, and symbol hiding from kallsyms.

🗂️

OverlayFS + TmpFS

OverlayFS, TmpFS XATTR, and POSIX ACL enabled for systemless module support compatible with Magisk, KernelSU modules, and APATCH.

🔎

Full KALLSYMS

Complete kernel symbol table exposed, required by Metamodule-style Magisk modules and kernel-level debugging tools.

📝

Signed builds

The 0001-scripts-sign-builds.patch patch ensures kernel images are signed, preventing verification failures on locked bootloaders during boot.

YASK — Automated Weekly Release

schedule · weekly workflow_dispatch 4-target matrix LTO full Wild-KSU canary

YASK (Yet Another Stock Kernel) is the flagship automated workflow. It triggers every Sunday at 00:00 UTC and builds all four kernel variants in parallel, packages them into a single AnyKernel3 ZIP, pushes it to a rolling GitHub Release, and sends a Telegram notification with build info and a banner image.

Trigger

.github/workflows/YASK.yml
on:
  schedule:
    - cron: '0 0 * * 0'    # Every Sunday 00:00 UTC
  workflow_dispatch:        # Manual trigger from GitHub UI

Pipeline — build-kernels job

✨ Setup workspace
Creates the per-matrix kernel directory, stamps BUILD_TIME env, installs repo rsync aria2.
📐 Setup SWAP 10 GB
Expands GitHub runner swap to 10 GB via pierotofy/set-swap-space, sets vm.swappiness=80.
🌟 Clone kernel source
Shallow clones the matrix kernel repo into its dedicated directory (--depth=1 --recursive).
📥 Initialize AOSP repo + toolchains
Syncs the AOSP kernel manifest, downloads clang-r547379, removes three unused Clang versions to reclaim disk space, patches build.config.constants with correct BRANCH and CLANG_VERSION, removes the defconfig check step.
🧩 Configure kernel & patches
Appends CONFIG_LOCALVERSION with the build date. Applies the GKI fixup patch, then loops through extra_patches defined per matrix entry (sign-builds, no-dirty, no-auto_localversion).
🔧 Add BBG (KSU variants only)
Runs the Baseband Guard setup script, enables CONFIG_BBG=y, and registers baseband_guard as a default LSM alongside SELinux in security/Kconfig.
🔧 Setup Wild-KSU + SUSFS (KSU variants only)
Installs Wild_KSU canary. Clones SUSFS4KSU and applies the kernel patch. Force-copies fs/susfs.c, include/linux/susfs.h, and susfs_def.h. Integrates susfs_init() into ksu.c (tries patch first, falls back to manual sed). Runs a full verification step and fails the build if anything is missing. Appends all SUSFS Kconfig options.
⚙️ Setup basic configs
Appends performance, TCP congestion, OverlayFS, TmpFS, and KALLSYMS config options to the defconfig.
📝 Patch Makefile
Appends -Wno-implicit-function-declaration to KBUILD_CFLAGS to suppress upstream warnings that would abort the build.
🔨 Build kernel
Invokes build/build.sh with LTO=full using all available CPU cores.
📤 Upload artifact
Uploads the built Image file as a GitHub Actions artifact (1-day retention) for the packaging job.

Pipeline — package-kernels job

📥 Download all kernel artifacts
Downloads all four Image files produced by the parallel matrix jobs.
📦 Create release package
Clones AnyKernel3 from the YASK branch, copies all Image files in, creates an ASCII banner, and zips everything into Anykernel3_YASK-WildKSU-AIO-FULL_YYYY-MM-DD.zip.
⬆️ Upload to GitHub Release
Uses softprops/action-gh-release to push the ZIP to the rolling YASK-NEXT release tag (requires GH_TOKEN secret).
📱 Telegram notification
Sends a sendPhoto call with the YASK banner image and a caption containing build metadata, supported devices, and a download link. Sends a failure message with log link if the job status is not success.

Main Build — Quick Manual Build

workflow_dispatch LTO thin Wild-KSU stable clang-r547379

The simplest manual workflow. Accepts a kernel source URL and branch, builds KSU, non-KSU, or both variants, and delivers a flashable ZIP to Telegram. Ideal for quickly testing a specific upstream branch or a custom kernel fork without modifying the automated YASK pipeline.

💡
Main uses Wild-KSU stable, making it the better choice when you want a production-ready KSU kernel rather than the bleeding-edge canary used by YASK and Advanced.

Pipeline Steps

✨ Setup workspace
Installs repo rsync aria2. Sets BUILD_TIME.
📐 Setup SWAP 10 GB
Same as YASK — expands runner swap with swappiness=80.
🌟 Clone kernel source
Clones from kernelSourceURL / kernelBranch inputs.
📥 Initialize repo
Syncs AOSP manifest, installs clang-r547379, removes three old Clang directories, patches build.config.constants, replaces DEFCONFIG in build.config.gki to use the user-supplied device defconfig, disables KMI enforcement in build.config.gki.aarch64.
⚙️ Configure kernel
Appends optional CONFIG_LOCALVERSION, CONFIG_KALLSYMS, CONFIG_TMPFS_XATTR, CONFIG_TMPFS_POSIX_ACL.
🔧 Setup Wild-KSU + SUSFS (ksu variant only)
Installs Wild_KSU stable. Applies SUSFS kernel patch, force-copies susfs source files, creates drivers/kernelsu symlink, appends all SUSFS Kconfig options. Does not include BBG (no BBG step in this workflow).
🔨 Build kernel
Calls build/build.sh with LTO=thin.
📦 Prepare artifact
Copies out/dist/Image to Image.noksu or Image.ksu depending on the matrix variant.
📦 Package + Telegram
Clones AnyKernel3 master branch, creates ZIP as Anykernel3_WildKSU_<branch>_<date>.zip, uploads as artifact (7 days), sends ZIP directly to Telegram as a document.
⚠️
Main Build does not include Baseband Guard (BBG). If you need BBG, use YASK or Advanced Build.

Advanced Build — Full Control

workflow_dispatch O2 / O3 / Os 100–1000 Hz 2 toolchain modes thin or full LTO

The most powerful workflow. Every significant build parameter is exposed as a GitHub Actions input. Supports two toolchain modes and four optimization levels, and provides the most detailed Telegram build summary.

Toolchain Modes

ModeClangSourceBuild command
android-tools r584948b github.com/topnotchfreaks/clang build/build.sh via AOSP repo
custom-tools greenforce latest greenforce-project/greenforce_clang (latest release) make LLVM=1 CC=clang directly

The custom-tools path fetches the latest release from the greenforce GitHub API at build time, meaning it always uses the newest available Clang. The android-tools path uses a pinned AOSP Clang version (r584948b) for reproducible builds.

Pipeline Steps

✨ Setup workspace
Installs an extended set of packages: repo rsync aria2 build-essential bc curl flex cpio dwarves libelf-dev gcc-aarch64-linux-gnu libssl-dev python3 clang lld llvm.
🌟 Clone kernel source
Clones from user-supplied URL and branch.
📥 Initialize repo (android-tools)
Syncs AOSP manifest, installs clang-r584948b (note: newer than r547379 used in main/YASK), removes more old Clangs, patches build configs, fully disables KMI enforcement including MODULES_ORDER and MODULES_LIST.
💫 Get Toolchains (custom-tools)
Queries GitHub releases API for greenforce-project/greenforce_clang, downloads the latest tar.gz, extracts it, locates the bin/ directory, exports CC AR AS NM OBJCOPY OBJDUMP STRIP, and validates that clang is in PATH.
📐 Setup SWAP 20 GB
Expanded to 20 GB (vs 10 GB in YASK/main) to accommodate aggressive O3/full-LTO builds.
😎 Generate make args (custom-tools only)
Builds a make argument string: -j$(nproc) O=out ARCH=arm64 LLVM=1 LTO=thin CROSS_COMPILE=aarch64-linux-gnu- etc., stored in step output for later steps.
📝 Patch Makefile
Adds four warning suppressions: -Wno-default-const-init-unsafe -Wno-uninitialized-const-pointer -Wno-implicit-function-declaration -Wno-sometimes-uninitialized (one more than YASK/main).
⚙️ Configure kernel
Handles LTO (thin/full with explicit Kconfig entries), optional localversion, KALLSYMS/TmpFS, optimization level (O2/O3/Os via CONFIG_CC_OPTIMIZE_FOR_*), tick rate (sets CONFIG_HZ and CONFIG_HZ_xxx, removing conflicting entries first), and applies the sign-builds patch.
🔧 Add BBG
Applied on all builds (not KSU-conditional as in YASK). BBG setup runs unconditionally.
🔧 Setup Wild-KSU + SUSFS (ksu variant)
Installs Wild_KSU canary. Applies SUSFS kernel patch. Force-copies SUSFS source files. Creates drivers/kernelsu symlink pointing to ../KernelSU (not ../Wild_KSU as in YASK). Appends SUSFS Kconfig options.
🔨 Build (android-tools)
Calls build/build.sh with LTO=thin or LTO=full based on input.
👍 Make defconfig + 🎶 Build (custom-tools)
Runs make <args> <DEFCONFIG_NAME> then make <args> LLVM=1 CC=clang ... with all LLVM tools explicitly passed.
📦 Package + Telegram
AnyKernel3 master branch, ZIP as Anykernel3_<branch>_<date>.zip (artifact, 1 day). Telegram sends ZIP with a rich caption including LTO, optimization, tick rate, and build type.
💡
Advanced Build uses a slightly newer Clang (r584948b) than YASK and Main (r547379) when in android-tools mode. This may produce marginally different code generation.

Main / Advanced — Dynamic matrix

Strategy matrix driven by buildKSU input
strategy:
  matrix:
    variant: ${{ fromJSON(
      # buildKSU == 'both'  → ["noksu", "ksu"]  # 2 parallel jobs
      # buildKSU == 'ksu'   → ["ksu"]            # 1 job
      # buildKSU == 'noksu' → ["noksu"]          # 1 job
    ) }}

Both workflows use fail-fast: false implicitly, so one variant failing does not cancel the other. Job names are prefixed with the variant: kernel-noksu and kernel-ksu.

KernelSU & SUSFS Integration

✅
All three workflows use Wild_KSU — an actively maintained KernelSU fork — combined with SUSFS for comprehensive root detection hiding at the kernel level.

Wild_KSU Installation

# Canary — YASK.yml and advanced_build.yml
curl -LSs "https://raw.githubusercontent.com/WildKernels/Wild_KSU/wild/kernel/setup.sh" | bash -s canary

# Stable — main.yml
curl -LSs "https://raw.githubusercontent.com/WildKernels/Wild_KSU/wild/kernel/setup.sh" | bash -s stable

SUSFS Integration Steps

1 · Apply kernel patch
50_add_susfs_in_gki-android13-5.15.patch — patches core VFS and process hooks required by SUSFS. If this patch fails, source files are still force-copied in the next step.
2 · Copy SUSFS source files
fs/susfs.c, include/linux/susfs.h, and include/linux/susfs_def.h are always force-copied into the kernel tree from the SUSFS4KSU repo.
3 · Integrate susfs_init() into ksu.c (YASK only)
YASK additionally applies 10_enable_susfs_for_ksu.patch to inject a susfs_init() call into Wild_KSU's initialization code. If the patch fails, it falls back to manual sed insertion. The result is verified — build fails if susfs_init or the include are missing.
4 · Append SUSFS Kconfig options
All feature flags are appended to the defconfig. See the table below for the full list.

Build config patches

After the toolchain is installed, build.config.constants is patched to reflect the correct branch and Clang version. build.config.gki has the hardcoded gki_defconfig replaced with the user-specified defconfig. build.config.gki.aarch64 has KMI enforcement disabled (YASK keeps enforcement on by not patching this file).

Secrets & Repository Tokens

⚠️
Configure these in GitHub → Repository → Settings → Secrets and variables → Actions → New repository secret before running any workflow.
GH_TOKEN
A GitHub Personal Access Token with repo scope. Used only by YASK.yml to push the AnyKernel3 ZIP to the YASK-NEXT GitHub Release via softprops/action-gh-release. Not required for main.yml or advanced_build.yml.
TELEGRAM_BOT_TOKEN
Telegram Bot API token obtained from @BotFather. Used by all three workflows to send build notifications via sendPhoto (YASK) or sendDocument (Main, Advanced).
TELEGRAM_USER_ID
Telegram chat ID of the recipient — either a personal user ID or a channel ID. The bot must be an admin of the channel if using a channel ID. Obtain your user ID from @userinfobot.
